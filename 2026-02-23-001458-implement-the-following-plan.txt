⏺ I've been using covy hands-on across all 14 commands now, so here are my
  honest opinions as an AI agent — the primary target audience for these
  changes.

  What's genuinely good

  Speed is a killer feature. 4-30ms for most commands means I can call covy
  multiple times in a single task without the user noticing. Most dev tools I
  interact with (linters, test runners, build systems) take seconds to minutes.
  Covy feels instant.

  The data model is well-designed. The ingest → state → query pipeline means I
  can run covy ingest once and then query the cached state with check, report,
  diff without re-parsing XML every time. That's the right architecture.

  The new JSON-by-default-when-piped behavior is exactly right. I was parsing
  ANSI table output before this change, which was fragile and ugly. Now I just
  pipe and get structured data.

  What still hurts me as an agent

  1. Flag discoverability is painful

  I hit wrong flags 4 times during testing:
  - --output vs --out vs --output-coverage vs --out-comment vs --out-sarif
  - covy pr --comment-out → actually --out-comment
  - covy ingest --out → actually --output
  - covy merge --out-coverage → actually --output-coverage

  There's no consistent convention. Some commands use --out, some --output, some
   --out-<thing>. I have to run --help for every command to discover the right
  flag. A consistent naming convention (--output everywhere, or --out 
  everywhere) would save me a round-trip every time.

  2. covy init writes to git root, not CWD

  This bit me. In a monorepo where JavaTest is a subdirectory, covy init writes
  covy.toml to the repo root, not the project directory. Most tools (like npm 
  init, cargo init) write to CWD. An agent working in a subdirectory has no way
  to create a project-scoped config without manually writing the file — which
  defeats the purpose of covy init.

  Suggestion: default to CWD, add --repo-root flag for the current behavior.

  3. No --json flag on most commands

  covy shard plan has --json. covy shard update has --json. But covy check, covy
   diff, covy doctor, covy map-paths use --report json or have no JSON mode at
  all. As an agent, I want one universal pattern:

  covy <anything> --json

  Right now I have to remember:
  - covy check → --report json
  - covy diff → --report json
  - covy shard plan → --json
  - covy impact → --json
  - covy doctor → no JSON mode at all
  - covy map-paths → no JSON mode at all

  covy doctor and covy map-paths having no machine-readable output means I have
  to regex-parse human text like "Mapped paths: 2/2 (100.0%)". That's fragile.

  4. Exit codes are inconsistent

  - covy check returns 0 (pass) or 1 (gate fail) — good
  - covy init returns 0 on success, 2 on error — good
  - But I can't distinguish "covy failed to run" (bad args, missing files) from
  "covy ran but the gate failed" without parsing the output. A convention like 0
   = success, 1 = gate/quality failure, 2 = usage/runtime error would be ideal
  (and it seems partially there already).

  5. covy ingest doesn't tell me what it wrote

  $ covy ingest jacoco.xml -q
  (no output)

  With -q I get nothing. Without -q I get tracing noise. What I actually want is
   a single JSON line like:
  {"files_ingested": 1, "coverage_files": 2, "output": ".covy/state/latest.bin"}

  6. The tracing/logging output goes to stdout, not stderr

  When I run covy check jacoco.xml --report json, the INFO lines and the JSON
  both go to stdout. That means if I'm parsing JSON output, the [INFO] 
  Ingesting... lines corrupt the JSON. I have to use -q to suppress them.
  Log/tracing output should always go to stderr so stdout is clean for machine
  consumption.

  7. No schema/help for JSON inputs

  The new error messages with examples are a big improvement. But ideally I'd
  want:
  covy shard plan --schema
  That prints the expected input JSON schema before I fail. Right now I have to
  intentionally fail to discover the format.

  8. Missing covy check --stdin for piped coverage data

  This exists but there's a gap — I can't pipe the output of one covy command
  into another easily. For example, covy merge writes a binary .bin file, and
  then I need covy check to read it. There's no covy check --input merged.bin
  (that's on covy diff). The UX for chaining commands is a bit inconsistent.
